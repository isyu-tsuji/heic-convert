# heic-flip 開発ルール

## 言語

- すべての応答・コード内コメント・コミットメッセージは日本語
- コミットタイトルは50文字以内、本文は72文字で折り返し

## 設計方針

- Functional Core / Imperative Shell を意識する
  - 純粋な変換ロジック（HEICデコード→画像エンコード）は副作用なしの関数として切り出す
  - File API, DOM操作, ダウンロードなどの副作用はShell層に隔離
- 境界バリデーション: 外部入力（ファイル、ユーザ入力）はZodで型検証
- Branded Types を活用し、`HeicFile` と `ConvertedBlob` などを混同しない型設計

## 技術スタック

- Vite + React + TypeScript
- libheif-js (WASM) でブラウザ内変換、サーバ送信は禁止
- ZIP化は jszip
- スタイルは未定（指示なければ最小限のCSS）

## 進め方

- 参考ドキュメント: プロジェクトの戦略・ロードマップは `ROADMAP.md` を参照
- 大きな変更（複数ファイルの新規追加、依存追加、アーキテクチャ変更）は実装前に計画を提示し、承認を取ること
- 計画には「目的・変更ファイル一覧・想定リスク」を含める
- 不確実な仕様判断は推測せず質問する

## やってはいけないこと

- ファイル内容をサーバに送信する処理を入れない
- 認証・課金・アナリティクス系のコードを勝手に追加しない
- 既存ファイルを大量に書き換えるリファクタを承認なしに行わない

## テスト

- 変換ロジック（Functional Core）には Vitest でユニットテストを書く
- UI層のテストはMVP段階では不要
